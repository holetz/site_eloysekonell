## Critical Issues

### OG cover image is missing from public/

- **Severity:** CRITICAL
- **Location:** `public/images/` (file absent), `src/layouts/Layout.astro:20`
- **Issue:** `Layout.astro` sets `DEFAULT_OG_IMAGE = "${SITE_URL}/images/og-cover.jpg"`, but `public/images/og-cover.jpg` does not exist. Every page that does not pass an explicit `ogImage` prop (home, `/sobre/`, `/metodologia/`, `/contato/`, `/faq/`, `/servicos/*`) will emit a broken OG image tag, resulting in no preview card when links are shared on LinkedIn, WhatsApp, or Twitter.
- **Fix:** Create `public/images/og-cover.jpg` at 1200×630 px. This is a single file addition with immediate impact on every share.

### Inconsistent JSON-LD `@id` fragment (slash vs no slash)

- **Severity:** HIGH
- **Location:** Multiple files — canonical definition in `src/layouts/Layout.astro:38,83` uses `#eloyse` / `#business` (no slash), while `src/layouts/CaseLayout.astro:58,59`, `src/pages/sobre.astro:16`, `src/pages/servicos/gestao-estrategica-de-pessoas.astro:107,110`, `src/pages/cases/index.astro:25,26`, `src/pages/servicos/index.astro:77`, `src/pages/servicos/mentoria-executiva.astro:60` use `/#eloyse` / `/#business` (with slash).
- **Issue:** Schema.org `@id` values must be identical strings for entity linking to work. Google's structured data parser treats `https://eloysekonell.com.br#eloyse` and `https://eloysekonell.com.br/#eloyse` as two different entities, breaking the knowledge graph connection between `Person`, `ProfessionalService`, and derived pages.
- **Fix:** Pick one canonical form (recommend `#eloyse` and `#business`, no slash, matching `Layout.astro`) and apply it uniformly to all files listed above.

## Technical Debt

### `SITE_URL` constant duplicated across 13+ files

- **Severity:** HIGH
- **Location:** `src/layouts/Layout.astro:15`, `src/layouts/BlogLayout.astro:39`, `src/layouts/CaseLayout.astro:39`, `src/components/BreadcrumbList.astro:8`, `src/pages/contato.astro:5`, `src/pages/sobre.astro:8`, `src/pages/faq.astro:5`, `src/pages/sitemap.xml.ts:3` (as `SITE`), plus all four `src/pages/servicos/*.astro` and both `src/pages/cases/*.astro`.
- **Issue:** The domain `https://eloysekonell.com.br` is hard-coded as a local constant in every file that needs it. A domain change or staging environment would require updating 13+ files. The canonical source of truth (`astro.config.mjs` has `site:`) is ignored.
- **Fix:** Create `src/lib/constants.ts` (or `src/lib/site.ts`) exporting `SITE_URL`, `SITE_NAME`, and `DEFAULT_OG_IMAGE`. Import from there. Alternatively, read from `import.meta.env.SITE` which Astro populates from `astro.config.mjs`.

### Hardcoded years of experience bypass `credenciais.ts`

- **Severity:** HIGH
- **Location:** `src/components/Hero.astro:15,35`, `src/components/About.astro:23,42`, `src/components/Consultoria.astro:16`, `src/pages/blog/index.astro:167`, `public/llms.txt:5`
- **Issue:** The single source of truth `src/data/credenciais.ts` defines `anosAtuacao: 18`, but components use "Há mais de 10 anos", "+10 anos", and "Há mais de uma década" as literal strings. `public/llms.txt` hard-codes "18 anos". When `credenciais.anosAtuacao` is updated, these components will not update automatically — the stat shown in the About section stats card (`src/components/About.astro:42`) already shows "10+" while `credenciais.ts` says 18. This is a live data inconsistency.
- **Fix:** Replace literal strings in components with `{credenciais.anosAtuacao}+` or `Há mais de {credenciais.anosAtuacao} anos`. `public/llms.txt` cannot be dynamically generated by Astro as-is — add a note or generate it as a `.astro` page at `/llms.txt`.

### `readingTime` is a manually maintained required field

- **Severity:** MEDIUM
- **Location:** `src/content/config.ts:14` (`readingTime: z.string()`), every `src/content/blog/*.md`
- **Issue:** Reading time is required in the schema but is typed as a free-form string (e.g., "8 min de leitura") that the author must estimate and maintain by hand. There is no automated calculation. If the post body is edited significantly, the reading time becomes inaccurate silently.
- **Fix:** Remove `readingTime` from the Zod schema and compute it at build time using word-count. Astro's `render()` returns the rendered content — count words from the markdown body and derive minutes. Alternatively mark it optional with a calculated fallback.

### `@astrojs/sitemap` installed but unused

- **Severity:** LOW
- **Location:** `package.json:13` (`@astrojs/sitemap: ^3.7.2`), `astro.config.mjs` (integration absent)
- **Issue:** The official `@astrojs/sitemap` integration is listed as a dependency but never imported or registered in `astro.config.mjs`. Instead, a fully custom sitemap is implemented at `src/pages/sitemap.xml.ts`. The unused package adds ~17 KB to `node_modules` and creates confusion about which sitemap generator is canonical.
- **Fix:** Remove `@astrojs/sitemap` from `package.json` (`npm uninstall @astrojs/sitemap`). The custom implementation in `src/pages/sitemap.xml.ts` is more capable for this project's needs.

### `cases` collection `getCollection` wrapped in try/catch with `as any`

- **Severity:** LOW
- **Location:** `src/pages/sitemap.xml.ts:79`
- **Issue:** The sitemap file wraps the cases collection query in a `try/catch` with a comment "Collection 'cases' not yet defined — skip silently", and casts the collection name `as any`. The collection is now fully defined in `src/content/config.ts:24`. This workaround is stale and suppresses potential real errors silently.
- **Fix:** Remove the try/catch and the `as any` cast. Use the typed `getCollection('cases', ({ data }) => !data.draft)` directly.

### Multiple uses of `any` type suppress TypeScript strictness

- **Severity:** MEDIUM
- **Location:** `src/layouts/BlogLayout.astro:49,51,53,191`, `src/pages/blog/index.astro:9,10,16,123`
- **Issue:** Despite `tsconfig.json` extending `astro/tsconfigs/strictest`, blog collection queries use `({ data }: any)`, `allPosts.find((p: any) => ...)`, and render variables typed as `any[]`. This defeats strict mode for the blog data layer and can mask type errors (e.g., accessing missing fields on post data).
- **Fix:** Use typed `CollectionEntry<'blog'>` (already imported in `src/pages/blog/[...slug].astro`) consistently across all blog-related files.

### `@/` path alias documented but unused

- **Severity:** LOW
- **Location:** `tsconfig.json:5-7`, `src/data/credenciais.ts:5` (comment only)
- **Issue:** `tsconfig.json` defines a path alias `@/*` mapping to `src/*`, and `credenciais.ts` mentions it in a comment. However, no file in the codebase actually uses `@/` imports — all imports use relative paths. The alias exists only in documentation, not in practice.
- **Fix:** Either adopt the alias consistently in new code (especially useful for deep imports like `../../data/credenciais`) or remove it from `tsconfig.json` to avoid confusion.

## Performance Concerns

### Global CSS has grown to 1,306 lines without scoping strategy

- **Severity:** MEDIUM
- **Location:** `src/styles/global.css`
- **Issue:** Nearly all component styles are in `global.css`, including component-specific selectors like `.hero`, `.about`, `.consultoria`, `.testimonials`, `.clients`, etc. The file is 1,306 lines. As the site grows, this file becomes the main performance bottleneck at build time and adds unused CSS to every page (e.g., blog-specific rules on the home page and vice versa).
- **Fix:** Gradually migrate component-specific styles into `<style>` blocks within each `.astro` component, allowing Astro to scope them and dead-code-eliminate per page. Start with the largest isolated sections: `.blog-main` styles currently in `global.css`.

### Large page files contain extensive scoped `<style>` blocks

- **Severity:** MEDIUM
- **Location:** `src/pages/blog/index.astro` (811 lines), `src/pages/servicos/assessment.astro` (718 lines), `src/pages/servicos/gestao-estrategica-de-pessoas.astro` (637 lines)
- **Issue:** Large page files mix content/logic with 400–600 lines of CSS. This makes the files hard to maintain and review. The blog index page alone is 811 lines, largely CSS. There is no consistent split between component markup and styles.
- **Fix:** Extract large `<style>` blocks into co-located CSS files or move them to component-level `.astro` files. No immediate performance risk (Astro scopes and minifies them), but a maintenance risk as the codebase grows.

### Unsplash-hosted blog cover images depend on external availability

- **Severity:** MEDIUM
- **Location:** `src/content/blog/lideranca-tecnica-vs-humana.md:7`, `src/content/blog/hora-de-mudar.md:8`, `src/content/blog/cadeira-vazia.md:8`, `src/content/blog/saude-mental-estrategia.md:8`, `src/content/blog/sucessao-mal-planejada.md:8`, `src/content/blog/identificar-potencial-de-lideranca.md:7`, `src/content/blog/feedback-continuo-90-dias.md:7`, `src/content/blog/tecnico-virou-gestor.md:8`, `src/content/blog/rh-futuro.md:8`, `src/content/blog/sucessao-empresa-familiar.md:7`, `src/content/blog/sucessor-e-sucessao.md:8`, `src/pages/blog/index.astro:253,268`
- **Issue:** Most blog cover images are hotlinked from `images.unsplash.com`. These are served at build time as `background-image: url(...)` CSS — not processed by Astro's image optimizer. If Unsplash changes a URL, enforces referrer policies, or goes down, all affected posts will show no cover image. OG images for social sharing depend on the same external URLs.
- **Fix:** Download cover images to `public/images/blog/` and update frontmatter to local paths. Use Astro's `<Image>` component or `background-image` with verified local assets. Priority: any posts used as `ogImage` values.

### Inline `clamp()` values not using the `--fs-*` token scale

- **Severity:** LOW
- **Location:** `src/pages/blog/index.astro` (8 occurrences), `src/pages/faq.astro:179,261`, `src/pages/sobre.astro:222`, `src/pages/servicos/*.astro` (several)
- **Issue:** The `global.css` defines a full `--fs-h1` through `--fs-eyebrow` token scale, but page-level `<style>` blocks frequently re-declare custom `font-size: clamp(...)` values instead of using the tokens. This creates visual inconsistency and means heading scale changes must be applied in many places.
- **Fix:** For new pages, always reference `var(--fs-h1)`, `var(--fs-h2)`, `var(--fs-h3)` from the token scale. Replace inline clamp values with token references where sizes overlap.

## SEO / Accessibility Gaps

### Landing page (`/`) has no `<main>` landmark element

- **Severity:** HIGH
- **Location:** `src/pages/index.astro`, `src/components/Hero.astro`, `src/components/About.astro` (and all landing sections)
- **Issue:** `src/pages/index.astro` renders Nav, Hero, Campanha, Testimonials, Clients, Cta, and Footer directly under `<body>` with no wrapping `<main>` element. All other page types (`PageLayout`, `BlogLayout`, `CaseLayout`, `blog/index.astro`) correctly use `<main>`. The missing landmark prevents screen reader users from navigating directly to main content and reduces accessibility score.
- **Fix:** Wrap all landing page sections (except Nav and Footer) in `<main id="main-content">` inside `src/pages/index.astro`.

### Skip navigation link is absent from all pages

- **Severity:** HIGH
- **Location:** `src/layouts/Layout.astro:199` (`<body>` opening)
- **Issue:** CLAUDE.md states "Skip link present in `Layout.astro` (#main-content)" but the actual `Layout.astro` contains no skip link. Keyboard users must tab through the full navigation on every page load before reaching content. The corresponding CSS rule is also absent from `global.css`.
- **Fix:** Add `<a href="#main-content" class="skip-link">Pular para o conteúdo</a>` as first child of `<body>` in `Layout.astro`, and add the id `main-content` to the `<main>` element in each layout. Add CSS: `.skip-link { position: absolute; left: -9999px; } .skip-link:focus { left: 0; z-index: var(--z-modal); }`.

### Twitter/X `twitter:creator` and `twitter:site` tags absent

- **Severity:** LOW
- **Location:** `src/layouts/Layout.astro:174-179`
- **Issue:** Twitter card meta includes `card`, `url`, `title`, `description`, `image`, but omits `twitter:creator` and `twitter:site`. Without these, posts shared on X lose attribution to the account, reducing trust signals for the card preview.
- **Fix:** Add `<meta name="twitter:site" content="@eloysekonell" />` and `<meta name="twitter:creator" content="@eloysekonell" />` if a Twitter/X handle exists, or omit if Eloyse does not use X.

### `FaqBlock.astro` references undefined CSS variables `--font-serif` and `--font-sans`

- **Severity:** MEDIUM
- **Location:** `src/components/FaqBlock.astro:84,118,137`, `src/pages/faq.astro:213,240`
- **Issue:** `FaqBlock` and `faq.astro` use `var(--font-serif, ...)` and `var(--font-sans, ...)` as CSS variables, but the design system defines them as `--serif` and `--sans` in `src/styles/global.css:13-14`. The fallback values (`'Cormorant Garamond', serif` and `'Manrope', sans-serif`) happen to match, so the visual result is correct, but any future renaming of `--font-serif`/`--font-sans` in a different project or if the fallback is removed will silently break typography in these components.
- **Fix:** Replace `var(--font-serif, ...)` with `var(--serif)` and `var(--font-sans, ...)` with `var(--sans)` in both files to align with the actual token names.

### Favicon is a full-size logo PNG, not a proper `.ico` or SVG

- **Severity:** LOW
- **Location:** `src/layouts/Layout.astro:181`
- **Issue:** The favicon uses `<link rel="icon" type="image/png" href="/images/logo.png">` pointing to the full navigation logo. This is a PNG optimised for display at ~140px wide. Browsers scale it down for favicon use (16px, 32px), which may produce poor quality. No `apple-touch-icon`, no `sizes` attributes, and no `.ico` fallback are present.
- **Fix:** Generate a `favicon.ico` (multi-size: 16, 32, 48 px) and a `favicon.svg` from the logo, place in `public/`, and add proper `<link>` tags. Add an `apple-touch-icon` at 180×180 px.

### `About.astro` stats do not use `credenciais.ts` — data inconsistency

- **Severity:** HIGH
- **Location:** `src/components/About.astro:41-57`
- **Issue:** The stats block in `About.astro` hardcodes `"10+"` years and uses non-numeric labels like `"Pipeline"` and `"Estratégia"`, which cannot be derived from `credenciais.ts`. Meanwhile, `src/pages/sobre.astro` correctly reads `credenciais.anosAtuacao` (18). The home page About section and the `/sobre` page show different numbers, creating a direct factual inconsistency visible to visitors.
- **Fix:** Import `credenciais` in `About.astro` and replace the `"10+"` literal with `{credenciais.anosAtuacao}+`. Align the stat labels to match `/sobre`.

## Security Concerns

### `navigator.clipboard` used without error handling

- **Severity:** LOW
- **Location:** `src/layouts/BlogLayout.astro:221`
- **Issue:** `navigator.clipboard.writeText(...)` is called without a `.catch()`. The Clipboard API fails silently in non-secure contexts (non-HTTPS) and requires user permission on some browsers. No fallback (e.g., `document.execCommand('copy')`) is provided. The site is served over HTTPS so this is low risk in production, but the user receives no feedback if the operation fails (the `title` attribute update at line 222 only triggers on success).
- **Fix:** Add `.catch(() => { /* optionally show an error indicator */ })` to the clipboard call.

### Hardcoded WhatsApp phone number in multiple locations

- **Severity:** LOW
- **Location:** `src/components/AssessmentSpotlight.astro:88,142`, `src/layouts/BlogLayout.astro:141`, `src/remark-blog-directives.mjs:96`, `src/pages/contato.astro` (implicit)
- **Issue:** The phone number `5547991443844` is embedded as a literal string in multiple components and in the remark plugin (which generates HTML for blog posts). If the contact number changes, it must be hunted down across source files, components, and the Markdown processing layer.
- **Fix:** Add `whatsapp: '+5547991443844'` to `src/data/credenciais.ts` and import it in components. The remark plugin cannot import TypeScript modules directly, so it should accept the number via a plugin option passed from `astro.config.mjs` using an env variable.

## Maintenance Risks

### CI/CD lacks a build failure notification mechanism

- **Severity:** MEDIUM
- **Location:** `.github/workflows/deploy.yml`
- **Issue:** The GitHub Actions workflow has no Slack, email, or GitHub notification step for build failures. If the Astro build fails (e.g., due to a broken frontmatter field in a new blog post), the deploy silently fails and the site is not updated. The repository owner may not notice until manually checking Actions.
- **Fix:** Add an `on-failure` step or use a GitHub notification action to alert on build failures. At minimum, GitHub's default email notifications for failed workflows should be confirmed enabled.

### No linting, formatting, or pre-commit hooks configured

- **Severity:** MEDIUM
- **Location:** `package.json:7-10` (scripts), root directory
- **Issue:** There is no ESLint, Prettier, Biome, or Stylelint configuration. There are no `lint` or `format` scripts in `package.json`. There are no pre-commit hooks (Husky, lint-staged). Code style is maintained entirely by convention, and inconsistencies already exist (e.g., some CSS uses minified single-line format in `global.css`, while page `<style>` blocks use expanded format).
- **Fix:** Add `@astrojs/eslint-config` or `eslint-plugin-astro` with a minimal config; add Prettier with a `.prettierrc`. Run them via `npm run lint` and `npm run format`. Optionally add `husky` + `lint-staged` to enforce on commit.

### No automated tests of any kind

- **Severity:** MEDIUM
- **Location:** Entire project — no `*.test.*` or `*.spec.*` files found
- **Issue:** There are zero tests: no unit tests for utilities (`credenciais.ts`, `remark-blog-directives.mjs`), no integration tests for routes, and no E2E tests for the contact flow or tab interaction. Regressions in structured data, sitemap generation, or the remark plugin go undetected until production.
- **Fix:** Minimum viable testing: unit tests for `remark-blog-directives.mjs` using Vitest (directive parsing is complex and has no safety net); a basic Playwright smoke test confirming the home page loads and the WhatsApp CTA is present.

### `devToolbar` is enabled in production builds

- **Severity:** LOW
- **Location:** `astro.config.mjs:14-16`
- **Issue:** `devToolbar: { enabled: true }` is set in the Astro config. Astro's dev toolbar only renders in dev mode and does not appear in production builds, so this has no direct user-facing impact. However, it is unusual to enable it explicitly in the main config (Astro enables it by default in dev anyway), and it signals the config was modified without a clear purpose.
- **Fix:** Remove the `devToolbar` block from `astro.config.mjs` to reduce noise.

### Date formatting is inconsistent: `timeZone: 'UTC'` only in `blog/index.astro`

- **Severity:** LOW
- **Location:** `src/pages/blog/index.astro:34` (uses `timeZone: 'UTC'`), `src/layouts/BlogLayout.astro:42-46`, `src/layouts/CaseLayout.astro:42-46` (no timezone specified)
- **Issue:** `blog/index.astro` explicitly passes `timeZone: 'UTC'` to `toLocaleDateString`, preventing timezone-shift bugs where a `pubDate` of 2026-04-15 displays as April 14 in UTC-3. The two layouts that format dates for the article header do not set `timeZone`, meaning dates displayed inside posts may appear one day earlier for Brazilian visitors.
- **Fix:** Add `timeZone: 'UTC'` to `toLocaleDateString` in `src/layouts/BlogLayout.astro:42-46` and `src/layouts/CaseLayout.astro:42-46`.

### `public/llms.txt` is a static file with hardcoded facts

- **Severity:** LOW
- **Location:** `public/llms.txt`
- **Issue:** `llms.txt` hardcodes "18 anos" and a manually-maintained list of pages. When `credenciais.anosAtuacao` is updated or a new page is added, `llms.txt` must be manually updated. It is out of sync with `credenciais.ts` (which is the declared source of truth) and with the actual site structure.
- **Fix:** Convert `public/llms.txt` to `src/pages/llms.txt.ts` (a GET endpoint returning plain text), importing data from `credenciais.ts` and auto-listing routes via `import.meta.glob`, similar to the sitemap pattern in `src/pages/sitemap.xml.ts`.

## Missing Infrastructure

### No error monitoring or analytics

- **Severity:** MEDIUM
- **Location:** Entire project
- **Issue:** There is no error tracking (Sentry, Bugsnag) and no analytics (Plausible, Fathom, Google Analytics). For a business site, it is impossible to know whether contact form submissions are failing, which pages receive traffic, or if 404 errors are common.
- **Fix:** Add privacy-friendly analytics (Plausible or Fathom, given the brand positioning) via a `<script>` in `Layout.astro`. Add a 404 page at `src/pages/404.astro` with a CTA back to home. Consider lightweight error monitoring for client-side script failures.

### No 404 page

- **Severity:** MEDIUM
- **Location:** `src/pages/` — no `404.astro` file
- **Issue:** Astro generates a basic fallback for unknown routes, but there is no custom `404.astro`. Visitors who follow broken links get a generic error page with no navigation, no branding, and no CTA to re-engage.
- **Fix:** Create `src/pages/404.astro` using `PageLayout` with a branded message, site navigation, and a link back to the home page or contact form.
